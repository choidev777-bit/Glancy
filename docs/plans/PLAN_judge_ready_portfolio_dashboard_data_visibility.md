## Overview

The current judge-facing portfolio dashboard does not yet satisfy the intended demo experience:

- The backend sample response contains 5-year fundamental history, but the browser can fall back to built-in sample data when `http://localhost:8000` times out, hiding the new history.
- The visible dashboard still shows `OHLCV 24개`, which is generated by the frontend fallback, not by rich uploaded/sample OHLCV.
- The technical analysis timeframe buttons (`1H`, `1D`, `1W`, `1M`) update state but do not change the displayed uploaded/sample candle series.

Target outcome: when a judge opens `http://localhost:5173/`, the first dashboard screen must show a self-contained composite portfolio demo for 삼성전자, AAPL, MSFT, SPY, BTC, GLD. Each asset must expose meaningful `요약 / 기술적 분석 / 기본적 분석`; technical analysis must have rich 1-year OHLCV and working timeframe aggregation; fundamental analysis must visibly show 2021-2025 history even if the backend sample API is slow or unavailable.

## Architecture Decisions

1. **Make the judge demo self-contained**
   - The dashboard must not depend on backend availability for core visible proof.
   - Keep API loading, but built-in fallback must contain the same richness as the sample API.
   - Before changing fallback behavior, verify whether `http://localhost:8000` is actually running and whether `VITE_API_BASE_URL` points to the intended backend.

2. **Store generated sample data as deterministic project data**
   - Add deterministic 1-year hourly OHLCV sample generation or committed generated data for the six-asset portfolio.
   - Avoid live network calls during judge demo.
   - Label the data as “샘플 API 기준 스냅샷” or “업로드 CSV 기준 스냅샷”, not real-time market data.
   - Do not call yfinance/Binance/Kiwoom from the first screen at runtime.
   - Recommended data volume targets:
     - 삼성전자/AAPL/MSFT/SPY/GLD: 1-year market-hours-like hourly candles, target 1,500-1,800 rows per asset, minimum 1,200.
     - BTC: 1-year 24/7 hourly candles, target 8,760 rows, minimum 4,000 if performance requires trimming.
   - If using CSV metric-row format, estimate row count before committing. OHLCV rows are `candles x 5 metrics`; a full target can exceed 80,000 CSV rows.

3. **Treat uploaded/sample OHLCV as the source of truth**
   - `1H`: raw hourly candles.
   - `1D`: aggregate by date: first open, max high, min low, last close, sum volume.
   - `1W`: aggregate by week.
   - `1M`: aggregate by month.
   - The selected timeframe must change the chart and the displayed OHLCV count.
   - For v1, aggregate uploaded/sample timestamps in UTC to avoid per-market timezone complexity. Keep the UI copy as snapshot analysis, not exchange-session official bars.

4. **Rename or clarify fundamental trend UX**
   - The tab currently says `분기 추이`, but data is annual 2021-2025.
   - Rename to `5년 추이` when annual history is supplied, or make the label generic as `추이`.

5. **Keep changes surgical**
   - Do not refactor unrelated asset-search pages.
   - Do not introduce a new charting library.
   - Do not add live data fetching to the judge demo path.

6. **Testing approach must match current tooling**
   - Current frontend package has no `test`, Vitest, or Jest setup.
   - Preferred option: add Vitest only if implementing pure helper tests for candle aggregation.
   - Conservative option: keep frontend verification to `npm run build` plus browser manual checklist.
   - The implementer must choose one explicitly before Phase 1 frontend test work begins. Recommended default: add Vitest for `src/lib/candle-timeframe` only, because timeframe aggregation is business logic and easy to test.

## Risks and Mitigations

| Risk | Probability | Impact | Mitigation |
| --- | --- | --- | --- |
| Large CSV slows sample API or frontend rendering | Medium | Medium | Use deterministic compact hourly data, test build and browser performance |
| Fallback and API data drift again | High | High | Route both through shared parser/normalizer or generated shared data |
| Timeframe aggregation creates incorrect OHLCV | Medium | High | Test open/high/low/close/volume aggregation explicitly |
| Annual fundamental data hidden behind misleading `분기 추이` label | High | Medium | Rename to `5년 추이` for annual labels |
| Backend timeout during judging | Medium | High | Rich fallback must be first-class and not look like an error |
| Frontend test tooling not present | High | Medium | Either add Vitest for pure helper tests or explicitly rely on build/manual checks |
| 1-year hourly CSV becomes large | Medium | Medium | Define row-count target, prefer deterministic generation/shared fixture if needed |
| Timezone differences make daily bars look odd | Medium | Low | Use UTC aggregation for v1 and label as snapshot analysis |

## Assumptions

- The judge-facing portfolio dashboard is a deterministic snapshot demo, not a live trading terminal.
- 1-year technical data can be generated/stored as sample data rather than fetched live during judging.
- The exact financial values are demo-safe illustrative values unless a later task adds a verified data collection pipeline.
- BTC uses crypto/market fundamentals, not corporate financial statement metrics.
- The implementation should remain surgical and avoid redesigning the entire dashboard.
- Frontend test tooling is not currently installed; adding Vitest is acceptable only for pure helper logic if the implementer chooses the recommended path.
